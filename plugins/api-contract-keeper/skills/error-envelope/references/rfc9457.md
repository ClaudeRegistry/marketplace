# RFC 9457, Problem Details for HTTP APIs

RFC 9457 ("Problem Details for HTTP APIs", 2023) is the successor to RFC 7807. The **wire format is unchanged**: same members, same `application/problem+json` media type, 9457 just clarifies extension registration and is the reference you should cite. If your code says "RFC 7807," it is still correct; prefer 9457 in new docs.

## The media type
Serve error bodies as `Content-Type: application/problem+json` (or `application/problem+xml`). This lets a client tell an error apart from a normal `application/json` payload without inspecting the body.

## Standard members
All are optional; pick a consistent subset and always include `type` and `status`.

| Member | Type | Rule |
|---|---|---|
| `type` | string (URI) | Identifies the problem **kind**. Default `about:blank` (then `title` should be the status phrase). When not `about:blank`, it SHOULD dereference to human-readable docs. This is what clients switch on. |
| `title` | string | Short, human summary of the `type`. Stable per `type`; does not change per occurrence. |
| `status` | number | The HTTP status, duplicated for intermediaries/logs that only have the body. Must equal the status line. |
| `detail` | string | Explanation for **this** occurrence. Safe for humans, no secrets, no stack traces. |
| `instance` | string (URI) | Identifies the specific occurrence (request path, or `/errors/{id}` you can look up in logs). |

## Extension members
Any additional top-level keys are allowed and are how you carry machine-readable detail. Common, recommended:
- `errors`: an array of per-field violations, `[{ "field": "email", "message": "must be a valid email", "code": "format" }]`. Use a `pointer` (JSON Pointer, e.g. `/items/0/qty`) for nested fields.
- `traceId` / `requestId`: correlate the response with server logs/tracing.
- `code`: your stable internal error code (e.g. `"order_already_paid"`), independent of the HTTP status.
- `retryAfter`: seconds, mirroring the `Retry-After` header on 429/503.

Extension members must not collide with the standard names, and a client that does not understand one must ignore it.

## Canonical examples

### Single validation failure
```http
HTTP/1.1 422 Unprocessable Content
Content-Type: application/problem+json

{
  "type": "https://api.example.com/problems/validation-error",
  "title": "Your request parameters didn't validate.",
  "status": 422,
  "detail": "The request body has 2 invalid fields.",
  "instance": "/orders",
  "traceId": "b1e5a0c2-4b2a-4f1e-9c3a-0f2b7d9e1a44",
  "errors": [
    { "field": "email", "message": "must be a valid email address", "code": "format" },
    { "field": "quantity", "message": "must be >= 1", "code": "min" }
  ]
}
```

### Business conflict (idempotency / state)
```http
HTTP/1.1 409 Conflict
Content-Type: application/problem+json

{
  "type": "https://api.example.com/problems/order-already-paid",
  "title": "Order already paid",
  "status": 409,
  "detail": "Order 8421 was already paid at 2026-07-01T10:22:03Z.",
  "instance": "/orders/8421/payments",
  "code": "order_already_paid"
}
```

### Minimal (about:blank)
```json
{ "type": "about:blank", "title": "Not Found", "status": 404 }
```
When `type` is `about:blank`, `title` SHOULD be the HTTP status phrase, clients that only know `about:blank` fall back to the status code.

## Multiple errors
RFC 9457 adds guidance for reporting several problems at once. Two accepted shapes:
1. **One problem with an `errors` extension** (preferred for validation), a single 422 whose `errors` array lists every field failure, as above. Return **all** field violations in one response, not just the first.
2. **A problem whose `errors` members are themselves problem objects**: for genuinely independent sub-operations (batch endpoints), each element carries its own `type`/`status`/`detail`.

## RFC 7807 → 9457 migration notes
- No wire change: existing 7807 producers/consumers are compliant with 9457.
- 9457 formalizes that **extension members are for machine consumption** and clarifies the registry for problem `type`s.
- Action item: update references and docs to 9457; keep your `type` URIs stable so clients don't break.

## Anti-patterns to reject in review
- **200-with-error**: `HTTP/1.1 200 OK` and `{ "success": false, "error": "..." }`. Breaks caches, retries, and every generic client. The status line must carry the error.
- **Envelope drift**: `{ "error": "x" }` on one route, `{ "message": "y" }` on another, `{ "errors": {...} }` on a third. Pick one shape everywhere.
- **Leaking internals**: stack traces, SQL, `SELECT ... FROM`, file paths, or ORM exception text in `detail`. Log server-side; return a `traceId`.
- **Status/body mismatch**: `status: 400` in the body but `500` on the wire (or vice versa).
- **Prose as the contract**: clients string-matching `detail`/`title`. Give them a stable `type` (and optional `code`) to branch on.
- **First-error-only validation**: returning after the first invalid field forces clients into fix-one-resubmit loops; aggregate all violations.
- **Wrong media type**: `application/json` on an error body, so clients can't distinguish it from success.
- **Wrong status semantics**: `400` for a semantic/validation failure (should be `422`), `403` for unauthenticated (should be `401`), `500` for a client mistake.
